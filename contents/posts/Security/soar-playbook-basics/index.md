## 플레이북이란?

SOAR에서 **플레이북(Playbook)** 은 보안 이벤트 발생 시 자동으로 실행되는 **대응 절차의 자동화 구현체**입니다.

기존에 분석가가 머릿속에 갖고 있던 대응 절차 — "피싱 메일이 오면 첨부파일 해시 추출 → VirusTotal 조회 → 악성이면 발신자 차단 → 티켓 생성" — 를 코드 없이 시각적으로 설계하고 실행할 수 있게 해줍니다.

플레이북이 잘 설계되면:
- 반복적인 단순 업무에서 분석가를 해방
- 24시간 일관된 대응 (사람이 자리를 비워도)
- 오탐(False Positive) 처리 시간 단축
- 대응 기록이 자동으로 남아 감사(audit) 용이

## 플레이북의 구성 요소

플레이북은 크게 4가지 요소로 이루어집니다.

```
┌─────────────────────────────────────────────────────┐
│                     Playbook                        │
│                                                     │
│  [Trigger] → [조건 분기] → [액션] → [액션] → ...   │
│                    ↓                                │
│               [다른 경로]                           │
└─────────────────────────────────────────────────────┘
```

### 1. 트리거 (Trigger)

플레이북을 **시작시키는 조건**입니다.

| 트리거 종류 | 예시 |
|------------|------|
| 레코드 생성 | 새 Alert가 생성될 때 |
| 레코드 수정 | Alert 상태가 "New"→"In Progress"로 바뀔 때 |
| 스케줄 | 매일 오전 9시 |
| 수동 | 분석가가 직접 버튼 클릭 |
| 웹훅(Webhook) | 외부 시스템이 이벤트를 전송할 때 |

실무에서 가장 많이 쓰는 트리거는 **"레코드 생성"** 입니다. SIEM에서 Alert가 들어오는 순간 자동으로 플레이북이 실행됩니다.

### 2. 액션 (Action)

실제로 **무언가를 하는 단계**입니다. 커넥터(Connector)를 통해 외부 시스템과 통신합니다.

```
액션 예시:
- VirusTotal에 해시 조회
- Jira에 티켓 생성
- Slack에 알림 전송
- 방화벽에 IP 차단 규칙 추가
- 이메일 발송
- 데이터 파싱 / 변환
```

### 3. 조건 (Condition)

**분기 처리**입니다. 조회 결과에 따라 다음 단계를 다르게 실행합니다.

```
예: VirusTotal 결과가 악성인가?
    ├── YES → IP 차단 + 심각도 "Critical"로 변경
    └── NO  → 분석가에게 검토 요청 알림
```

### 4. 반복문 (Loop)

여러 개의 항목을 **순회 처리**합니다.

```
예: Alert에 포함된 IP 목록 전체를 순회하며
    각각 위협 인텔리전스 조회 실행
```

## 실전 시나리오: 피싱 이메일 자동 분석

가장 흔한 보안 시나리오로 플레이북 설계를 이해해 봅시다.

### 상황

- 직원이 의심스러운 이메일을 신고
- 분석가는 첨부파일과 URL을 수동으로 확인해야 함
- 하루에 수십 건 들어오면 지침

### 자동화 목표

신고 이메일이 접수되면:
1. 첨부파일 해시, URL 자동 추출
2. VirusTotal / 위협 인텔리전스 자동 조회
3. 악성 판정 시 → 발신자 도메인 차단 + 티켓 생성 + 팀 알림
4. 정상 판정 시 → 오탐 처리 + 신고자에게 안내 메일

### 플레이북 흐름도

```
[Trigger: 피싱 신고 Alert 생성]
        ↓
[이메일 원본에서 IOC 추출]
(첨부파일 해시, URL, 발신자 도메인)
        ↓
[Loop: IOC 목록 순회]
        ↓
[VirusTotal API 조회]
        ↓
[조건: 악성 탐지 비율 > 5/70?]
   ↓ YES              ↓ NO
[발신자 도메인 차단]  [다음 IOC로]
[Jira 티켓 생성]
[Slack 팀 알림]
        ↓
[Loop 종료 후]
[악성 IOC 있었나?]
   ↓ YES              ↓ NO
[Alert 심각도 High]  [오탐 처리]
[신고자에게 결과 메일] [신고자에게 안내]
```

이 플레이북 하나로 분석가가 30분 걸리던 작업을 **3분 안에** 처리합니다.

## 좋은 플레이북 설계 원칙

### 1. 원자적(Atomic) 액션 단위로 설계

플레이북 하나가 너무 크면 유지보수가 어렵습니다. "IP 위협 조회", "티켓 생성", "알림 발송"처럼 작은 단위로 쪼갠 뒤 **서브 플레이북**으로 호출하는 방식이 좋습니다.

```
메인 플레이북
├── 서브 PB: IOC 추출
├── 서브 PB: 위협 인텔리전스 조회  ← 재사용 가능
├── 서브 PB: 티켓 생성             ← 재사용 가능
└── 서브 PB: 알림 발송             ← 재사용 가능
```

### 2. 오류 처리 필수

외부 API가 다운될 수도 있습니다. 각 액션에 타임아웃과 실패 경로를 반드시 설계합니다.

```
[VirusTotal 조회]
    ↓ 성공        ↓ 실패 (타임아웃/오류)
[결과 처리]     [분석가에게 수동 검토 요청]
                [에러 로그 기록]
```

### 3. 사람 개입(Human Task) 포인트 설계

모든 것을 자동화하면 안 됩니다. **고위험 액션**(계정 비활성화, 호스트 격리 등)은 분석가 승인 후 실행하도록 설계합니다.

```
[악성 판정: 심각도 Critical]
        ↓
[분석가에게 호스트 격리 승인 요청]
        ↓ 승인           ↓ 거부
[호스트 격리 실행]    [재검토 태스크 생성]
```

### 4. 멱등성(Idempotency) 고려

같은 Alert로 플레이북이 중복 실행될 수 있습니다. "이미 차단된 IP인지 확인 후 차단"처럼, 같은 액션이 두 번 실행되어도 문제없도록 설계합니다.

## 플레이북 vs 런북(Runbook)

혼용되는 경우가 많지만 차이가 있습니다.

| 구분 | 런북(Runbook) | 플레이북(Playbook) |
|------|--------------|------------------|
| 형태 | 문서 (PDF, Wiki) | 자동화 워크플로우 |
| 실행 | 사람이 읽고 따라함 | 시스템이 자동 실행 |
| 유연성 | 높음 (상황 판단 가능) | 설계된 경로만 |
| 속도 | 느림 | 빠름 |
| 역할 | 의사결정 가이드 | 반복 업무 자동화 |

이상적으로는 런북을 기반으로 플레이북을 만들고, 자동화하기 어려운 판단은 런북으로 남겨두는 방식입니다.

## FortiSOAR 플레이북 실제 화면 구조

FortiSOAR를 기준으로 실제 플레이북 구성 요소를 살펴봅니다.

```
[Start]
  │
  ├── Trigger: On Create of Alert
  │
  └─→ [Set Variable]
        alert_id = {{alert.id}}
        severity = {{alert.severity}}
        │
        └─→ [Connector: VirusTotal]
              Action: Get IP Reputation
              Input: {{alert.sourceIP}}
              │
              ├── [Condition]
              │   malicious_count > 5?
              │     │
              │     ├── YES → [Update Alert]
              │     │         severity = "Critical"
              │     │         status = "Investigating"
              │     │           │
              │     │           └─→ [Create Ticket in Jira]
              │     │               [Send Slack Notification]
              │     │
              │     └── NO  → [Update Alert]
              │               status = "False Positive"
              │
              └─→ [End]
```

FortiSOAR에서는 이 구조를 드래그&드롭으로 시각적으로 설계하고, 각 스텝에서 Jinja2 템플릿 문법으로 데이터를 참조합니다:

```jinja2
{{vars.steps.Get_IP_Reputation.data.positives}}
{{alert.sourceIP}}
{{globalVars.Server_fqdn}}
```

## 플레이북 개발 시 자주 하는 실수

**1. 예외 처리 없이 배포**
API 실패, 빈 값 반환 등을 처리하지 않으면 플레이북이 중간에 멈춥니다. 모든 조회 액션에 실패 경로를 추가하세요.

**2. 하드코딩된 값**
IP, URL, API 키를 플레이북 안에 직접 쓰면 유지보수가 어렵습니다. 전역 변수(Global Variable)나 파라미터를 활용하세요.

**3. 로그 없는 자동화**
자동화가 잘못 동작해도 추적이 안 됩니다. 주요 분기마다 로그 액션을 추가해 두면 디버깅이 쉬워집니다.

**4. 너무 많은 것을 한 플레이북에**
200개 스텝짜리 플레이북은 수정도, 테스트도 어렵습니다. 서브 플레이북으로 분리하세요.
